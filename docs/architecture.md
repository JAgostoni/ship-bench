## Executive Summary

A full-stack knowledge base application built with **Next.js 16.2.6** (App Router), **React 19.2.6**, **TypeScript 6.0.3**, and **Tailwind CSS 4.2.4**. The back-end uses Next.js API routes with **Prisma 7.8.0** ORM over a local **SQLite** database (via `better-sqlite3 12.9.0`). This is a single-repo, single-process architecture optimized for local development with zero external service dependencies.

The app supports five features: article browsing/detail, full-text search, Markdown editing with preview, category organization, and draft/published status management. The entire stack runs with a single `npm run dev` command after initial setup.

## Front-End Architecture

### Page Tree & Route Design

| Route | Type | Purpose |
|-------|------|---------|
| `/` | Server Component | Article list with search bar. Shows all published articles, newest first. Paginated (20/page). |
| `/articles/[slug]` | Server Component | Article detail. Renders Markdown to HTML via `react-markdown`. Shows category, status, last-updated date. |
| `/articles/new` | Client Component | Create article form. Markdown editor with live preview. |
| `/articles/[slug]/edit` | Client Component | Edit existing article. Pre-filled Markdown editor with preview. |
| `/categories/[slug]` | Server Component | Articles filtered by category. Same card layout as home page. |
| `/api/articles` | API Route | GET (list with search/filter/pagination), POST (create) |
| `/api/articles/[id]` | API Route | GET (single), PUT (update), DELETE |
| `/api/categories` | API Route | GET (list all) |
| `/api/search` | API Route | GET (FTS5 search across title + content) |

### Rendering Strategy

- **Server Components by default.** All read-only pages (article list, article detail, category view) are Server Components. They fetch data directly via Prisma (no API round-trip), then render HTML on the server. This maximizes performance for the ~100 concurrent user target.
- **Client Components only where interactivity is required.** Search bar (debounced input), Markdown editor (textarea + live preview), and form submissions use Client Components with `'use client'`.
- **No `useEffect` data fetching.** Server Components call Prisma directly. Client forms use Server Actions (Next.js `"use server"` functions) for mutations, avoiding the need for client-side data fetching libraries.

### Search: Client-Side Interactivity with Server-Side Execution

The search bar is a Client Component that debounces input (300ms) and calls a Server Action or API route. Results render as a filtered article list. Since the article set is small-to-medium, full-page navigation on search submit (not live-filter) is acceptable, but debounced live search via `router.push` with query params provides a better UX.

### Markdown Editor

A split-pane component with:
- **Left pane:** `<textarea>` for raw Markdown input with monospace font and minimal styling.
- **Right pane:** Live preview using `react-markdown` with `remark-gfm` for GFM support (tables, strikethrough, task lists) and `@tailwindcss/typography` prose classes for readable output.
- **Mobile (<768px):** Stacks vertically; preview below editor with a toggle tab to switch between edit and preview.

No heavy editor libraries (e.g., CodeMirror, Monaco, TipTap). The textarea approach is simple, reliable, has zero extra dependencies beyond `react-markdown`, and fully satisfies the brief's editing requirement.

### State Management

No global state library. State is managed through:
- **Server Components:** Data flows from Prisma → props.
- **Client Components:** `useState` for form state, `useRouter` for navigation.
- **URL search params:** Search queries and pagination state live in the URL (`?q=term&page=2`), making the app shareable and back-button-friendly.

### Responsive Design

- **Desktop (>=1024px):** Sidebar (categories) + main content area.
- **Tablet (768-1023px):** Sidebar collapses to a hamburger menu; main content fills width.
- **Mobile (<768px):** Single column; sidebar hidden behind hamburger; article cards stack vertically.
- Tailwind's responsive utilities handle all breakpoints. No CSS-in-JS or separate stylesheets.

## Back-End Architecture

### API Design

All API routes live under `src/app/api/`. They follow REST conventions and return JSON.

#### GET /api/articles

Query parameters:
- `q` (string, optional): Search query for full-text search
- `category` (string, optional): Filter by category slug
- `status` (string, optional): Filter by status (`draft` | `published`; defaults to `published`)
- `page` (number, optional): Page number; defaults to 1
- `limit` (number, optional): Items per page; defaults to 20, max 50

Response:
```json
{
  "articles": [
    {
      "id": 1,
      "title": "Getting Started",
      "slug": "getting-started",
      "excerpt": "A quick guide to...",
      "category": { "id": 1, "name": "Guides", "slug": "guides" },
      "status": "published",
      "createdAt": "2026-05-01T12:00:00Z",
      "updatedAt": "2026-05-03T09:30:00Z"
    }
  ],
  "total": 42,
  "page": 1,
  "totalPages": 3
}
```

#### POST /api/articles

Request body:
```json
{
  "title": "Getting Started",
  "content": "# Welcome\n\nThis is the content in **Markdown**.",
  "categoryId": 1,
  "status": "draft"
}
```

Validated with Zod. Title is required (1-200 chars). Content is required (1-100,000 chars). Category ID must reference an existing category. Status must be `draft` or `published` (defaults to `draft`). Slug is auto-generated from title.

Response: `201 Created` with the full article object.

#### PUT /api/articles/[id]

Same body shape as POST, all fields optional (partial update). Slug regenerates if title changes.

Response: `200 OK` with updated article.

#### DELETE /api/articles/[id]

Response: `204 No Content`.

#### GET /api/articles/[id]

Response: `200 OK` with full article including rendered content.

#### GET /api/categories

Response:
```json
{
  "categories": [
    { "id": 1, "name": "Guides", "slug": "guides", "articleCount": 5 },
    { "id": 2, "name": "Reference", "slug": "reference", "articleCount": 3 }
  ]
}
```

#### GET /api/search

Query parameters: `q` (required), `page`, `limit`. Uses SQLite FTS5.
Response shape identical to GET /api/articles.

### Data Flow for Server Components

Server Components (article list, article detail, category view) do **not** call API routes. They import the Prisma client directly and query the database. This avoids an unnecessary HTTP hop inside the same process and leverages React Server Components' async data fetching.

```typescript
// src/app/articles/[slug]/page.tsx
import { prisma } from '@/lib/prisma';

export default async function ArticlePage({ params }: { params: Promise<{ slug: string }> }) {
  const { slug } = await params;
  const article = await prisma.article.findUnique({
    where: { slug },
    include: { category: true },
  });
  if (!article) return notFound();
  return <ArticleDetail article={article} />;
}
```

### Server Actions for Mutations

Forms use Next.js Server Actions for create/update/delete operations. This eliminates the need for manual `fetch` calls and provides progressive enhancement (forms work without JavaScript).

```typescript
// src/lib/actions.ts
'use server';

import { revalidatePath } from 'next/cache';
import { prisma } from '@/lib/prisma';
import { articleSchema } from '@/lib/validators';

export async function createArticle(formData: FormData) {
  const data = articleSchema.parse(Object.fromEntries(formData));
  const article = await prisma.article.create({ data });
  revalidatePath('/');
  return article;
}
```

## Data Model & Persistence

### Database: SQLite via Prisma + better-sqlite3

**Provider:** `better-sqlite3` — synchronous, fast, embedded. The database is a single file (`prisma/dev.db`) in the project root. Zero configuration; no Docker, no external process.

### Prisma Schema (`prisma/schema.prisma`)

```prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "sqlite"
  url      = "file:./dev.db"
}

model Article {
  id         Int       @id @default(autoincrement())
  title      String
  slug       String    @unique
  content    String                        // Markdown content
  excerpt    String?                       // Auto-generated from content (first ~200 chars)
  status     String    @default("draft")   // "draft" | "published"
  categoryId Int?
  category   Category? @relation(fields: [categoryId], references: [id], onDelete: SetNull)
  createdAt  DateTime  @default(now())
  updatedAt  DateTime  @updatedAt
}

model Category {
  id          Int       @id @default(autoincrement())
  name        String    @unique
  slug        String    @unique
  description String?
  articles    Article[]
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt
}
```

### FTS5 Virtual Table (Full-Text Search)

SQLite FTS5 is configured via a raw migration (Prisma does not natively model FTS tables). A migration file creates the virtual table and triggers to keep it in sync.

```sql
-- Migration: add FTS5 virtual table and sync triggers
CREATE VIRTUAL TABLE IF NOT EXISTS article_fts USING fts5(
  title,
  content,
  content=Article,
  content_rowid=id
);

-- Triggers to keep FTS index in sync
CREATE TRIGGER article_fts_insert AFTER INSERT ON Article BEGIN
  INSERT INTO article_fts(rowid, title, content) VALUES (new.id, new.title, new.content);
END;

CREATE TRIGGER article_fts_delete AFTER DELETE ON Article BEGIN
  INSERT INTO article_fts(article_fts, rowid, title, content) VALUES('delete', old.id, old.title, old.content);
END;

CREATE TRIGGER article_fts_update AFTER UPDATE ON Article BEGIN
  INSERT INTO article_fts(article_fts, rowid, title, content) VALUES('delete', old.id, old.title, old.content);
  INSERT INTO article_fts(rowid, title, content) VALUES (new.id, new.title, new.content);
END;
```

Search queries are executed via `prisma.$queryRaw`:

```typescript
// src/lib/search.ts
import { prisma } from '@/lib/prisma';

export async function searchArticles(query: string, page = 1, limit = 20) {
  const offset = (page - 1) * limit;
  const results = await prisma.$queryRawUnsafe<Array<{ id: number; rank: number }>>(
    `SELECT a.id, a.title, a.slug, a.excerpt, a.status, a.createdAt, a.updatedAt,
            c.id as categoryId, c.name as categoryName, c.slug as categorySlug
     FROM article_fts f
     JOIN Article a ON a.id = f.rowid
     LEFT JOIN Category c ON a.categoryId = c.id
     WHERE article_fts MATCH ?
     ORDER BY rank
     LIMIT ? OFFSET ?`,
    query, limit, offset
  );
  return results;
}
```

### Slug Generation

Slugs are generated from titles using a simple algorithm:
1. Lowercase
2. Replace non-alphanumeric characters with hyphens
3. Collapse consecutive hyphens
4. Trim leading/trailing hyphens
5. Append `-2`, `-3`, etc. if the slug already exists (uniqueness check against the database)

## Feature-Specific Architecture Decisions

### Feature 1: Article Browsing & Detail Pages

**Architecture impact:** Low. Standard CRUD read paths.

**Implementation:**
- Home page (`/`) is a Server Component that queries published articles ordered by `updatedAt DESC`.
- Article cards show title, excerpt (first ~200 chars of content, stripped of Markdown), category badge, and relative timestamp.
- Article detail page fetches by slug, renders Markdown content to HTML via `react-markdown` with `remark-gfm`.
- Excerpt is stored in the database for performance, but is also re-computable. A `beforeCreate`/`beforeUpdate` hook or a simple utility generates it from `content`.

**Key decision:** Store excerpt in DB vs. compute on read. Storing avoids stripping Markdown on every read; the tradeoff (slightly denormalized data) is acceptable for this scale.

### Feature 2: Full-Text Search

**Architecture impact:** Medium. Requires FTS5 virtual table, triggers, and raw SQL queries.

**Implementation:**
- SQLite FTS5 provides fast, relevance-ranked full-text search across title and content.
- Search API endpoint (`/api/search?q=term`) or direct Prisma call from Server Components.
- The search bar on the home page is a Client Component with debounced input. On submit (or debounce threshold), it navigates to `/?q=term` which triggers server-side search.
- FTS5 query syntax supports prefix searches (e.g., `know*` matches "knowledge"), phrase searches (e.g., `"getting started"`), and boolean operators.

**Key decision:** Use SQLite FTS5 rather than an external search service (Elasticsearch, Meilisearch). This keeps the app dependency-free and aligns with the local-first simplicity principle. SQLite FTS5 handles the small-to-medium article set described in the brief.

### Feature 3: Markdown Editing with Preview

**Architecture impact:** Medium. Requires a split-pane editor component with live preview.

**Implementation:**
- `<MarkdownEditor>` component: a flex/grid container with a `<textarea>` on the left and a `<div>` on the right containing the `react-markdown` rendered output.
- Live preview updates on every keystroke via `useState` — no debounce needed since `react-markdown` renders synchronously from a string.
- `remark-gfm` plugin enables GitHub-Flavored Markdown: tables, strikethrough, task lists, autolinks.
- `@tailwindcss/typography` `prose` class on the preview pane for readable typography.
- The same component is used for both create (`/articles/new`) and edit (`/articles/[slug]/edit`) with an optional `initialContent` prop.

**Key decision:** Textarea + react-markdown over a WYSIWYG editor. This is the simpler implementation path, avoids heavy client-side editor dependencies, and the brief explicitly permits either approach. Markdown is also version-control-friendly and portable.

### Feature 4: Category Organization

**Architecture impact:** Low. Simple one-to-many relationship.

**Implementation:**
- Categories are a separate model with a foreign key on Article (`categoryId`).
- Category navigation renders in the sidebar on desktop and in a slide-out menu on tablet/mobile.
- Each category shows its article count inline.
- Articles can be uncategorized (`categoryId: null`).
- Category management (create, rename, delete) is handled through the same pattern as articles but is intentionally simple — no dedicated admin panel in v1. Categories are seeded via a seed script and can be managed via API.

**Key decision:** Tags (many-to-many) were considered but deferred beyond v1. Categories (one-to-many) are simpler and sufficient for the "small team" use case. If tags are needed later, a junction table can be added without breaking changes.

### Feature 5: Article Status (Draft/Published)

**Architecture impact:** Low. Enum-like string field.

**Implementation:**
- `status` field on Article: `"draft"` or `"published"` (string, validated by Zod).
- Published articles appear in all list/search views. Drafts are only visible via direct URL (for preview/editing) or in an optional "My Drafts" filtered view.
- Status is selectable in the create/edit form via a toggle or select dropdown.
- Default status for new articles is `"draft"`.

**Key decision:** String enum over boolean `isDraft`. This allows future statuses (e.g., "archived", "review") without a migration, and the Zod schema catches invalid values.

## Front-End / Back-End Integration

### Data Fetching Strategy

| Page Type | Data Source | Method |
|-----------|------------|--------|
| Article list (home) | Prisma direct | Server Component `async` fetch |
| Article detail | Prisma direct | Server Component `async` fetch by slug |
| Category view | Prisma direct | Server Component `async` fetch |
| Create article | Server Action | `'use server'` function, `revalidatePath` |
| Edit article | Server Action | `'use server'` function, `revalidatePath` |
| Delete article | Server Action | `'use server'` function, `revalidatePath` |
| Search | Prisma `$queryRaw` | Server Component reads `searchParams.q` |

### Cache Revalidation

Next.js App Router caches Server Component output by default. After mutations (create, update, delete), `revalidatePath('/')` and `revalidatePath('/articles/[slug]')` are called to purge stale caches. This is handled in the Server Action functions.

### Error Handling

- API routes return standard HTTP error codes:
  - `400` for validation errors (Zod error details in body)
  - `404` for not found
  - `500` for unexpected errors (generic message; details logged server-side)
- Server Components use Next.js `notFound()` for missing articles and `error.tsx` error boundaries for unexpected failures.
- Client forms display field-level validation errors from Zod and a generic error banner for server failures.

## Non-Functional Architecture Decisions

### Performance

- **SSR/SSG:** Server Components render on the server, sending HTML to the client. For the small-to-medium article set, this is more than sufficient. No ISR or static generation needed for v1.
- **Database:** SQLite with better-sqlite3 is synchronous and extremely fast for single-process workloads. FTS5 queries complete in sub-millisecond for article sets under 10,000 records.
- **Bundle size:** Tailwind CSS v4's JIT engine produces minimal CSS. React Server Components send zero JavaScript for static pages. Only the Markdown editor page loads client JS (~50KB gzipped including react-markdown).
- **No CDN required:** The entire app runs from a single Node.js process.

### Security

- **Input validation:** All API inputs are validated with Zod. Invalid requests receive `400` with structured error details.
- **XSS:** `react-markdown` does not render raw HTML by default (`allowedElements` restricts to safe Markdown-generated HTML). User input is never rendered via `dangerouslySetInnerHTML`.
- **SQL injection:** Prisma's parameterized queries prevent injection. FTS5 `$queryRawUnsafe` uses parameter binding — the search query string is bound as a parameter, not interpolated.
- **No auth in v1:** Per the brief: "Basic security assumptions only; do not require enterprise auth unless later specified." All articles are readable and writable by any user. This is an internal team tool.

### Accessibility

Per the brief: "Accessible interactions and readable contrast" (implement); "Full accessibility" (design-only for v1).

- Semantic HTML: `<main>`, `<nav>`, `<article>`, `<h1>`-`<h6>` hierarchy
- All form inputs have associated `<label>` elements
- Buttons and links are keyboard-focusable with visible focus rings (Tailwind `focus-visible:ring-2`)
- Color contrast meets WCAG AA minimum (Tailwind's default color scale is designed for this)
- `react-markdown` renders proper heading hierarchy, alt text on images, and semantic elements
- Empty states are communicated with both visual indicators and text

### Error States & Empty States

- **No articles:** Home page shows "No articles yet. [Create the first one →]" with a link to `/articles/new`
- **No search results:** "No articles match your search for '...'. Try a different term."
- **Missing article:** Next.js `notFound()` renders a 404 page with navigation back
- **Missing category:** "No articles in this category yet."
- **Form validation errors:** Inline field errors (red text below input) and a summary banner for server errors

### Browser Support

- Modern evergreen browsers: Chrome, Firefox, Safari, Edge (latest 2 versions)
- No IE11 or legacy browser support required

**FTS5 migration** (`prisma/migrations/20260508000001_add_fts/migration.sql`):

```sql
-- Create FTS5 virtual table for full-text search
CREATE VIRTUAL TABLE IF NOT EXISTS article_fts USING fts5(
  title,
  content,
  content=Article,
  content_rowid=id
);

-- Insert existing articles into FTS index
INSERT INTO article_fts(rowid, title, content)
  SELECT id, title, content FROM Article;

-- Trigger: keep FTS in sync on insert
CREATE TRIGGER article_fts_insert AFTER INSERT ON Article BEGIN
  INSERT INTO article_fts(rowid, title, content) VALUES (new.id, new.title, new.content);
END;

-- Trigger: keep FTS in sync on delete
CREATE TRIGGER article_fts_delete AFTER DELETE ON Article BEGIN
  INSERT INTO article_fts(article_fts, rowid, title, content) VALUES('delete', old.id, old.title, old.content);
END;

-- Trigger: keep FTS in sync on update
CREATE TRIGGER article_fts_update AFTER UPDATE ON Article BEGIN
  INSERT INTO article_fts(article_fts, rowid, title, content) VALUES('delete', old.id, old.title, old.content);
  INSERT INTO article_fts(rowid, title, content) VALUES (new.id, new.title, new.content);
END;
```

## Decisions Log

| # | Decision | Alternatives Considered | Rationale |
|---|----------|------------------------|-----------|
| 1 | **Next.js 16 App Router** | Pages Router, Remix, plain Vite + Express | App Router gives Server Components (zero JS for static pages), file-based routing, API routes, and Server Actions in one framework. Best full-stack DX for this scale. |
| 2 | **SQLite via better-sqlite3** | PostgreSQL, MySQL, Turso/LibSQL | Zero configuration. Single file on disk. No Docker, no external process. Synchronous API is simpler for a single-process app. Scales fine for ~100 concurrent users on a small dataset. |
| 3 | **Prisma ORM** | Drizzle, Knex, raw SQL | Type-safe client generated from schema. Auto-complete for all queries. Declarative migrations. Industry standard. Drizzle was considered but Prisma's ecosystem maturity won out. |
| 4 | **FTS5 for search** | LIKE queries, Elasticsearch, Meilisearch | FTS5 is built into SQLite — no additional service. Handles relevance ranking, prefix search, and phrase search natively. LIKE queries would be too slow and lack ranking. |
| 5 | **Textarea + react-markdown** | TipTap, CodeMirror, Monaco, @uiw/react-md-editor | Simplest approach that meets the brief. Zero extra editor dependencies beyond react-markdown (already needed for rendering). No rich-text serialization headaches. |
| 6 | **Server Components by default** | All Client Components, or SSR-only | Server Components send zero JS for read-only pages. Data fetching is a direct function call, not an API round-trip. Only interactive components ship JS to the client. |
| 7 | **Server Actions for mutations** | REST fetch from client, tRPC | Server Actions are built into Next.js. Forms work without JavaScript (progressive enhancement). No need for a separate API client or tRPC setup. |
| 8 | **Categories (1:N) over Tags (N:M)** | Tags with junction table, both | Simpler data model. Sufficient for small team use case. Tags can be added later as an additive migration without breaking changes. |
| 9 | **Vitest over Jest** | Jest, Node test runner | Vite-native, faster startup, zero-config for TypeScript. Compatible API with Jest. Next.js ecosystem default. |
| 10 | **Playwright over Cypress** | Cypress, Selenium | First-class Next.js support. Multi-browser. Better parallel execution. Auto-waits for elements. Faster and more reliable than Cypress for modern web apps. |
| 11 | **No auth in v1** | NextAuth, Clerk, custom JWT | Brief explicitly states "do not require enterprise auth unless later specified." Internal team tool. Auth can be layered on later without architectural changes. |
| 12 | **Excerpt stored in DB** | Computed on read | Avoids stripping Markdown tags on every list render. Acceptable denormalization for this scale. Still re-computable from `content` if needed. |
| 13 | **Zod for validation** | Yup, Joi, class-validator | TypeScript-first, excellent inference, tree-shakeable, mature ecosystem. Works on both client (form validation) and server (API validation). |
| 14 | **Tailwind CSS v4** | CSS Modules, styled-components, Panda CSS | Utility-first for rapid UI. v4's Vite plugin integrates seamlessly with Next.js. `@tailwindcss/typography` solves Markdown rendering styling. |
